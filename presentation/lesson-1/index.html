Overview

What is involved when you visit a website?

  • Web browser (Chrome, Firefox, Not Internet Explorer)
  • DNS lookup ("Domain Name System")
  • HTTP request sent
  • Server processes request
  • Response sent back to browser
  • Browser handles response

- This is roughly what is involved when you visit a website.

- Browser: Where you start, by typing a URL into the address bar.

- DNS: Like a phone book. Converts nice name to IP address.

- HTTP: (Hypertext transfer protocol). A format for transferring data.

- Server: A computer somewhere in the world, receives a message asking for a page and thinks about it.

- Response: Server decides what data to send back - often the HTML for a web page.

- Browser: Gets the response and decides what to do with it.

Web Browsers

Responsibilities

  • Handle DNS lookups to route traffic to IP addresses
  • Generate HTTP requests, and then later handles the response
  • Render HTML and CSS (Chrome uses Blink/Webkit), and run JavaScript (Chrome uses V8 engine)
  • Part of rendering an HTML document is to retrieve any additional files requested such as images, css, js
  • JavaScript can be used to dynamically create additional requests. This is how APIs function. (Asynchronous)

Will come back to all of these in more detail, so bear with me.

DNS: (DNS like a phonebook, except for IP addresses.) Web browser first of all hits the DNS server to ask for an IP address.

HTTP: Sends message to IP address saying "Hey give me this page". If server responds with "Well okay then!" -- [next fragment].

Render: If we did request a webpage, "render" it - ie, display it, in all it's colourful glory. As an FED I don't really know/care about how Blink/Webkit/V8 work beyond basics.

Additional requests: HTML is just text with instructions. Some instructions are to get images or CSS. When the browser requests CSS and JS it is synchronous. Images calls are async (they happen separate to the page loading).

AJAX: When data is sent or received via javascript, it's called asynchronous request. Happens in the background. Does the same kind of HTTP request the browser did.

Example: DisplaySweet.com

DNS Servers

Domain Name System

  • Convert nice looking "www.facebook.com" into a hard to remember IP address
  • IP addresses often change, like if you were to change hosting provider
  • There can be a delay in a DNS server being updated, making a website unreachable for a time
  • Browsers/operating systems will often cache a DNS entry so sometimes a device will lose a website for a while

Example diagram of how a request works: http://royal.pingdom.com/2009/06/08/a-visual-explanation-of-how-dns-lookups-work/

GET: type of request (come back to this in next slide) | filepath | protocol (HTTP v1.1)

Host: DNS has already looked up the IP for this. But the server wants it too so it can handle multiple websites or subdomains on one IP address.

Connection: "Don't go away, talk to you in a minute"

Pragma/Cache-control: Pragma was replaced, but serves same purpose. "Give me a fresh copy of this".

User-Agent: details about your browser and what you're viewing on - becomes useful for keeping users logged in.

Accept: "I want you to GET code-club.website.co.uk, and I will accept "text/html".

Cookies: snippets of text that are saved in the browser and accessible only by the website that set them.

We send a chunk of text, we get back a chunk of text.

JavaScript can make these requests too, in the background (async).

Request types

  • GET (Read)
  • POST (Create)
  • PUT (Update)
  • DELETE (Delete)

This is known as CRUD - Create, Read, Update, Delete.

The crux of how APIs work.

Old school sites will do these requests syncronously (causes page reload)

Ever seen the Chrome warning when you hit "back" - "Are you sure you want to resubmit this data?"

New sites (last 10 years) will mostly use asyncronous requests with JS

Web Servers

  • Linux (CentOS, RedHat, Ubuntu) or Windows Server
  • Apache, NGINX, Node.js
  • PHP, Ruby, Node.js, Python, C*
  • MySQL, MongoDB, PostgreSQL

Linux: OS designed to run programs for handling web stuff

Apache: handles HTTP requests and responses - "If requesting page /home, do this thing and send it back."

PHP: Processing data after a request

DBs: Storage of that data

Responses

Response codes

  • 2xx (Success)
  • 3xx (Redirection)
  • 4xx (Client error)
  • 5xx (Server error)

HTTP, 200, All good!

Cache nonsense: Instructions on how long the browser should cache for.

Content-Type: "Here's the format I returned it in"

ETag: Fingerprint to help with caching. Will remain the same if the file hasn't changed.

Accept-Ranges: Simply used for helping a server pause/continue file downloads. Not important.

Cache-Control: Server is telling us not to cache this, make sure we get a fresh one every time.

Vary: More caching stuff. I dunno, lot a crap here that isn't very interesting.

Content-Encoding: gzip is compression, so just informing the browser to unzip it first.

Content-Length: Size in bytes of the response.

Keep-Alive: Information about how long to wait responses here before giving up.

Content-Type: what sort of info did we get back? An HTML file to render? Some data?

Basically just information for the browser on how to interperet the data.

In addition: the response "body" - which would be the actual HTML file.
